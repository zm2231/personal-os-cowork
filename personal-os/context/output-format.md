### Core Note Format

Every file Claude writes to Obsidian uses this frontmatter. Adjust fields based on active plugins (see below).

```markdown
---
date: YYYY-MM-DD
type: session-summary
tags:
  - personalos
  - [skill-name]
related: []
source: PersonalOS
---

# [Title]

[Content in the skill's standard output format]

---
*Generated by /[skill-name] on YYYY-MM-DD*
```

### Plugin Compatibility

**Dataview** (if active):
- Use YAML frontmatter (above) — Dataview queries it with `WHERE type = "meeting-notes"` etc.
- Add any fields the user's existing Dataview queries expect — ask during onboarding: "Do you have Dataview queries that expect specific frontmatter fields? I'll match them."
- Avoid inline `field:: value` syntax unless user explicitly uses it — YAML frontmatter scales better
- Key Dataview-queryable fields to always include: `date`, `type`, `tags`, `source`

**Templater** (if active):
- Do NOT write Templater syntax (`<% %>`) into output files — that's for template files, not outputs
- If user has a Templater template for a note type (e.g., a meeting notes template), trigger it via Obsidian URI or ask user to apply it manually; then append content to the created file
- Alternative: write plain markdown output files that happen to match the structure of the user's templates
- Store template paths in this file under `## Templater Templates` if confirmed

**Tasks plugin** (if active):
- Format action items using Tasks-compatible syntax: `- [ ] Task description 📅 YYYY-MM-DD`
- Due date emoji `📅` is the Tasks plugin's due date marker
- Scheduled date: `⏳ YYYY-MM-DD` | Start date: `🛫 YYYY-MM-DD` | Priority: `🔺` (urgent) `⏫` (high) `🔽` (low)
- Example: `- [ ] Review PR #234 📅 2026-04-10 ⏫`

**Obsidian Properties** (Obsidian 1.4+):
- The YAML frontmatter above renders as "Properties" in the UI — no changes needed, it's the same format
- Date fields: use ISO format `YYYY-MM-DD` (Obsidian auto-recognizes as date type)
- Tags: use array syntax (shown above), not inline `#tag` in frontmatter

### How to Write (MCP tools)

**Append to existing page:**
```
mcp__notion__notion_from_title → get page ID by title
mcp__notion__notion_append_markdown → append content
```

**Create new page:**
```
mcp__notion__notion_create_page → parent_id = configured parent page
```

**Find a database:**
```
mcp__notion__notion_list_databases → find by name
mcp__notion__notion_create_database_item → add row
```

### Page Structure

Every Notion page Claude creates must have:
- **Title:** `[Skill Name] — [Topic] — YYYY-MM-DD`
- **Content:** skill output in markdown (Notion renders it)
- **Footer:** `*Generated by PersonalOS /[skill-name]*`

For database entries (e.g., Daily Huddle, task tracker), match the existing schema — read the database properties first before writing.

### Scalability Rules
1. Never create top-level pages — always nest under the configured parent
2. Before creating, check if a page with today's date already exists (append instead)
3. For recurring outputs (daily agenda, weekly summary), use a consistent title pattern so they stack cleanly
4. Tag pages with `PersonalOS` so they're filterable
